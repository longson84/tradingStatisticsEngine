## Problem Statement

The Trading Statistics Engine has accumulated significant architectural debt across
every layer — not just the Streamlit UI, but the naming conventions, module
organization, and the coupling between computation and rendering. Incremental
refactoring keeps producing more mess, not less. The correct path is a clean-room
rebuild of the backend as a proper Python library, decoupled from any framework.

The existing user-facing results are correct. The domain logic (strategies, trade
simulation, factor analysis) works. The problem is entirely in how the code is
organized, named, and coupled together.

## What Makes This Cool

A standalone Python library — `trading_engine` — that:
- Works in Jupyter notebooks, CLI scripts, and a web app equally well
- Has a clean, opinionated domain model: the library IS the documentation
- Makes FastAPI a thin adapter (< 300 lines), not a load-bearing layer
- Forces every abstraction to earn its place (no UI to hide behind)
- Means the React frontend is built against a real contract, not Streamlit scaffolding

The "library-first" constraint is the discipline that makes the rest of the stack clean.

## Constraints

## Domain Model

The library is organized into 6 layers. Each layer depends only on layers above it.

```
┌─────────────────────────────────────────────┐
│ 1. Data                                     │
│    DataLoader → PriceFrame (OHLCV unified)  │
├─────────────────────────────────────────────┤
│ 2. Factor                                   │
│    Factor(prices) → FactorSeries            │
│    (absorbs current indicators/ + factors/) │
├─────────────────────────────────────────────┤
│ 3. Factor Analysis                          │
│    Time-series: 1 symbol × 1 factor × time │
│    → FactorAnalysisResult                   │
│    Cross-sectional: N symbols × 1 factor   │
│    → CrossSectionalResult (breadth, ranks) │
├─────────────────────────────────────────────┤
│ 4. Strategy                                 │
│    Strategy(symbols, prices)                │
│    → WeightMatrix + TradeList               │
│    WeightMatrix: shape (time × symbols)     │
│    Values in [-1, 1]: long, flat, short     │
├─────────────────────────────────────────────┤
│ 5. Portfolio                                │
│    Portfolio(initial_capital, strategy)     │
│    → EquityCurve (NAV-based simulation)     │
│    Handles long + short P&L correctly       │
├─────────────────────────────────────────────┤
│ 6. Performance                              │
│    PerformanceAnalyzer(equity_curve, trades)│
│    → PerformanceReport (metrics, drawdown,  │
│       monthly returns, per-trade analysis)  │
└─────────────────────────────────────────────┘
```

### Layer 1: Data

```python
class DataLoader(Protocol):
    def load(self, symbol: str, start: date, end: date) -> PriceFrame: ...

@dataclass
class PriceFrame:
    symbol: str
    data: pd.DataFrame  # columns: open, high, low, close, volume
    source: str         # "yfinance" | "vnstock" | "csv"
```

Sources: yfinance, vnstock, CSV — each implements `DataLoader`.

### Layer 2: Factor

Absorbs current `indicators/` and `factors/` packages. A Factor is any function
from prices → derived series. Technical indicators (MA, Bollinger, Donchian) and
analytical factors (rarity score, AHR999) are all Factors.

```python
class Factor(Protocol):
    def compute(self, prices: PriceFrame) -> FactorSeries: ...

@dataclass
class FactorSeries:
    name: str
    values: pd.Series
    metadata: dict  # any additional context (e.g., window, params)
```

### Layer 3: Factor Analysis

Pure functions over `FactorSeries`. No Factor methods — this keeps analysis
logic decoupled from factor computation. Two modes, operating on different axes.

#### Time-series mode
One symbol, one factor, over time. Answers: "how has this factor behaved historically?"

```python
def percentile_breakdown(factor: FactorSeries, buckets: list[float]) -> FactorAnalysisResult
def rarity_analysis(factor: FactorSeries) -> FactorAnalysisResult
```

#### Cross-sectional mode
Many symbols, one factor, at each point in time. Answers: "how does this factor
distribute across a universe right now?" Produces market breadth indicators,
universe rankings, and threshold statistics.

```python
def analyze_cross_section(
    factor: Factor,
    universe: list[str],
    prices: dict[str, PriceFrame],
    threshold: float | None = None,
) -> CrossSectionalResult

@dataclass
class CrossSectionalResult:
    factor_name: str
    universe: list[str]
    # time-indexed series across all symbols:
    counts_above: pd.Series      # count(factor > threshold) at each t
    pct_above: pd.Series         # % of universe above threshold at each t
    breadth: pd.Series           # normalized [0, 1], same as pct_above / 100
    ranks: pd.DataFrame          # shape (time × symbols), rank per symbol per t
    universe_median: pd.Series   # median factor value across universe at each t
```

Example — "% of symbols with price below MA50":
```python
factor = MovingAverageRatio(window=50)   # computes MA50 / price
result = analyze_cross_section(
    factor=factor,
    universe=["BTC", "ETH", "SOL", "BNB"],
    prices=price_frames,
    threshold=1.0,   # MA50/price > 1 means price below MA50
)
# result.pct_above["2024-01-15"] → 0.75 (3 of 4 symbols below MA50)
# result.breadth → pd.Series usable as a regime indicator
```

Cross-sectional breadth feeds naturally into Layer 4: a strategy can consume
`CrossSectionalResult.breadth` as an input signal — e.g., reduce weights when
fewer than 30% of the universe is above a key moving average.

### Layer 4: Strategy

The core architectural upgrade. Strategy outputs a **weight matrix** — not binary
buy/sell signals. Weights are continuous values in `[-1.0, 1.0]`:
- `1.0`  = 100% long in this symbol
- `0.0`  = no position (flat)
- `-1.0` = 100% short in this symbol
- `0.5`  = 50% long

```python
class Strategy(Protocol):
    def compute(
        self,
        symbols: list[str],
        prices: dict[str, PriceFrame],
        regime: RegimeSeries | None = None,
    ) -> StrategyOutput: ...

@dataclass
class StrategyOutput:
    weights: pd.DataFrame    # shape: (time × symbols), values in [-1, 1]
    trades: list[Trade]      # derived from weight transitions

@dataclass
class WeightEvent:
    date: date
    weight: float
    price: float

@dataclass
class ExplainRecord:
    active_factors: dict[str, float]   # factor_name → value at entry
    breadth_at_entry: float            # pct of universe above threshold
    regime_at_entry: str               # "risk_on" | "risk_off" | "transition"

@dataclass
class Trade:
    symbol: str
    direction: Literal["long", "short"]  # explicit — do not derive from sign(weight)
    entry_date: date
    entry_price: float
    entry_weight: float                  # weight at open of the trade
    exit_date: date | None               # None if still open
    exit_price: float | None
    weight_history: list[WeightEvent]    # captures scaling in/out within the trade
    return_pct: float | None             # portfolio-weighted realized P&L
    holding_days: int | None
    mae_pct: float | None                # max adverse excursion
    mfe_pct: float | None                # max favorable excursion
    explain: ExplainRecord | None        # None unless factor context passed
```

Weight transitions define trades:
- `0 → positive`: long entry
- `positive → 0`: long exit
- `0 → negative`: short entry
- `negative → 0`: short exit
- `positive → negative` (or vice versa): exit + reverse **atomically** (2 records, same bar)
- `0.5 → 0.3` (same direction, non-zero): append to `weight_history`, no new trade record

**BaseStrategy pattern:** Strategy implementations only implement `_compute_weights()`.
The `BaseStrategy.compute()` base method auto-calls `_compute_weights()` then
`strategy/utils.py:weight_transitions_to_trades()` — no duplication across implementations.

```python
class BaseStrategy:
    def compute(self, symbols, prices, regime=None) -> StrategyOutput:
        weights = self._compute_weights(symbols, prices, regime)
        trades = weight_transitions_to_trades(weights, prices)
        return StrategyOutput(weights=weights, trades=trades)

    def _compute_weights(self, symbols, prices, regime) -> pd.DataFrame:
        raise NotImplementedError
```

### Layer 5: Portfolio

Takes `initial_capital` and strategy weights, runs NAV-based simulation.
Handles both long and short P&L correctly:
- Long P&L = weight × (price_t / price_{t-1} - 1)
- Short P&L = -weight × (price_t / price_{t-1} - 1)

```python
@dataclass
class Portfolio:
    initial_capital: float
    strategy: Strategy
    max_leverage: float = 1.0              # default: no leverage
    regime_config: RegimeConfig | None = None  # optional regime filter

def run_portfolio(portfolio: Portfolio, prices: dict[str, PriceFrame]) -> PortfolioResult

@dataclass
class PortfolioResult:
    equity_curve: pd.Series   # NAV over time
    trades: list[Trade]
    weights: pd.DataFrame     # the weight matrix that was applied
```

### Layer 6: Performance

Pure functions over `PortfolioResult`. No methods — all analysis is functional.

def analyze_performance(result: PortfolioResult) -> PerformanceReport

@dataclass
class PerformanceReport:

    sharpe_ratio: float

## CEO Review Decisions (2026-03-19)

### Accepted Expansions (added to scope)
- **Regime detection:** `detect_regime(breadth_series, thresholds) → RegimeSeries` in Layer 3.
  Labels each period as 'risk_on' | 'risk_off' | 'transition'.
  Wired via `run_portfolio(regime_config=RegimeConfig(...))` — not inside the strategy.
- **Trade explainability:** Optional `ExplainRecord` on `Trade`.
  Fields: `active_factors: dict[str, float]`, `breadth_at_entry: float`, `regime_at_entry: str`.
  Populated when caller passes factor context; `None` otherwise.
- **Strategy combiners:** Two meta-strategies implementing the Strategy Protocol:
  - `EnsembleStrategy(strategies, weights)` — weighted average of weight outputs
  - `RotationStrategy(strategies, selector)` — picks best strategy per period
  Both generate `trades` from their own averaged/selected weight transitions only.
- **Unified comparison framework:** `run_comparison([BacktestConfig]) → ComparisonReport`
  covers batch (vary symbols), sweep (vary params), multi-strategy (vary strategy).
  `ComparisonReport` has `results: list[PortfolioResult]` + `errors: list[ConfigError]`.
  Partial failure: collect errors, return what succeeded. Never fail silently.
- **`trading_engine.diagnostics.assert_no_leakage()`** — look-ahead bias detector.
  Ships in Phase 1. Verifies strategy never uses future price data.

### Deferred to TODOS
- **Live trading / Executor layer** — Layer 7, PaperTrader + broker adapters (Alpaca etc).
  Natural Phase 4 after library + FastAPI + React ship.

### Pre-build Decisions Locked
1. **Library location:** `trading_engine/` at repo top level (sibling to `src/`).
   `src/` is the legacy Streamlit app being phased out.
2. **Zero-crossing rule:** Atomic — `weight 0.5 → -0.3` generates 2 trade records
   (exit long + enter short) in the same bar.
3. **Comparison framework API:** Unified `run_comparison([BacktestConfig]) → ComparisonReport`.
   `BacktestConfig` holds (strategy, symbols, params). One function, N configurations.

### Error Handling Rules
- No `except Exception` or bare `except` anywhere in the library
- Named custom exceptions: `DataLoadError`, `FactorComputeError`,
  `InsufficientDataError`, `StrategyOutputError`, `ConfigError`
- `run_comparison` partial failure: collect errors, never raise for individual config failure
- All DataLoader calls: retry 2x on timeout, raise `DataLoadError` after
- NaN in weight output: always raise `StrategyOutputError` — never pass NaN downstream

## Eng Review Decisions (2026-03-19)

All decisions locked during `/plan-eng-review`. Build sequence below is authoritative.

1. **`max_leverage` + `regime_config`** — both on `Portfolio` dataclass (not on strategy).
   Strategy expresses conviction. Portfolio enforces capital constraints + regime filter.

2. **`direction` field kept on `Trade`** — explicit over clever.
   `direction: Literal["long", "short"]` is redundant with `sign(entry_weight)` but
   prevents subtle bugs and improves readability.

3. **`weight_history: list[WeightEvent]`** on `Trade` — captures scaling in/out.
   Partial weight changes (e.g., `0.5 → 0.3`) append a `WeightEvent`; no new trade record.
   Full zero-crossings produce new trade records (atomic rule).

4. **`BacktestConfig` has no `DataLoader`; includes date range.**
   `run_comparison(configs, prices)` — caller pre-fetches. Avoids N network calls for N configs.

5. **Centralized `trading_engine/types.py`** — all shared types in one file.
   FastAPI, tests, and all layers import from here. Built before any layer implementation.

6. **`BaseStrategy` auto-call pattern** — `weight_transitions_to_trades()` lives in
   `strategy/utils.py` and is called automatically by `BaseStrategy.compute()`.
   Implementations only override `_compute_weights()`.

7. **Per-layer build sequence locked** — Layer 1 → 2 → 3 → 4 → 5 → 6 → Diagnostics.
   Each layer has a validation gate (import smoke test + key tests) before proceeding.
   Tests written from scratch against new API; old tests kept as behavioral reference only.

## Open Questions

*All resolved as of 2026-03-19.*

1. **Capital constraint / leverage:** ✅ RESOLVED
   Leverage is allowed. `Portfolio` enforces it via `max_leverage: float = 1.0`.
   If `sum(abs(weights_t)) > max_leverage`, all weights are scaled proportionally
   so `sum(abs) = max_leverage`. Default = 1.0 (no leverage).
   Strategy stays decoupled from capital limits — it expresses conviction,
   Portfolio enforces constraints.

2. **Rebalancing frequency:** ✅ RESOLVED
   Daily re-evaluation at close. Weight matrix is date-indexed (daily bars).
   Portfolio rebalances at each day's closing price.

3. **Comparison framework data fetching:** ✅ RESOLVED
   Option B — caller pre-fetches. Signature:
   `run_comparison(configs: list[BacktestConfig], prices: dict[str, PriceFrame]) → ComparisonReport`
   `BacktestConfig` holds (strategy, symbols, params) — no DataLoader.
   FastAPI route pre-fetches + caches before calling run_comparison.
   Reason: a 50-config parameter sweep needs only 1 network call, not 50.
